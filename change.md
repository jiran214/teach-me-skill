# Teach Skill 改动方案

## 一、改造目标

将原 Teach Skill 从“以 HTML 课程文件为核心的教学系统”，改造成“以持续对话为核心、以用户上传资料为知识来源、由单一教师 Agent 完成讲解、练习、评估和教学调整的长期学习系统”。

核心变化：

```text
原版：
收集外部资料
→ 生成 HTML Lesson
→ 用户阅读和完成页面练习

改造后：
用户上传学习资料
→ Agent 解析并建立可渐进读取的资料库
→ 教师通过对话讲解
→ 使用 AskUserQuestion 或终端任务练习
→ 教师评估掌握情况
→ 未掌握则继续教学
→ 记录重要学习结果
```

HTML 不再是主要教学载体，只作为对话中的可选辅助工具。

---

## 二、最终的用户项目工作区结构

```text
workspace/
  MISSION.md
  TEACHER.md
  NOTES.md

  sources/
    book-a.pdf

  library/
    outline.md
    chunks/
        0001.md
        0002.md
        0003.md

  learning-records/
    0001-<dash-case-name>.md
    0002-<dash-case-name>.md

  assets/
    images/
    diagrams/
    html/
    scripts/
```

注意区分，该工作空间为skill执行时，用户的工作空间，而非本项目的目录结构，本项目的目录遵循skill格式

### 文件职责

`MISSION.md`

保存用户为什么学习、希望达到什么能力、当前学习范围和现实目标。所有教学决策必须围绕 Mission 展开。

`TEACHER.md`

保存教师的人设和互动方式，由用户定义或与用户共同创建。

包括：

- 教师名字
- 性格
- 语气
- 与学生的关系
- 讲解风格
- 鼓励方式
- 批评方式
- 对话长度偏好
- 是否喜欢使用比喻、故事、案例
- 互动边界

教师人设只影响表达方式，不能降低知识准确性和掌握标准。

`NOTES.md`

保存轻量、可变化的教学上下文：

- 用户的学习偏好
- 用户不喜欢的教学方式
- 当前正在学习的位置
- 下一次应该继续的内容
- 临时观察
- 资料处理状态
- 当前使用的教材

不要把完整会话记录写入 `NOTES.md`。

`sources/`

保存用户上传的原始资料，包括 PDF、Markdown、网页存档、图片或其他文档。

这些资料是教学的主要知识来源。

不要默认依赖模型的参数知识代替用户资料。资料缺失时，可以明确说明并询问用户是否允许补充外部来源。

`library/outline.md`

整个资料库唯一的导航入口。

它应包含pdf解析的所有一级章节、二级章节的目录，及每个章节详细内容的chunk文本路径
由脚本解析后生成

`library/chunks/`

保存从用户资料中切分出的内容。Chunk 应尽量遵循文档自然结构，不要只按固定 token 数量机械切分。

每个 chunk 至少保留：

```yaml
source: sources/example.pdf
title: 第二章 核心方法
section: 2.1 基本原理
pages: 27-31
previous: xxx.md
next: xxx.md
```

`learning-records/`

沿用原 Teach Skill 的学习记录机制。

只记录重要、稳定、会影响后续教学的学习结果，例如：

- 用户真正理解了某个关键概念
- 用户存在一个持续性误区
- 用户完成了一次有代表性的应用
- 用户的学习目标发生变化
- 用户表现出新的前置知识缺口
- 某种讲解方式对用户特别有效或无效

不要每次对话都创建记录。

记录示例：

```md
# Dense 不代表向量中没有零

## 背景

用户最初认为 dense vector 的含义是向量中不存在零值。

## 当前理解

用户已经理解，dense 表示大量维度共同承载语义，与稀疏表示的特征表达方式不同。

## 掌握证据

用户能够独立比较 dense retrieval 与 BM25 的表示差异。

## 尚未掌握

用户还不能解释向量相似度如何对应查询与文档的相关性。

## 对后续教学的影响

下一步应从几何直觉和训练目标解释向量相似度。
```

`assets/`

保存教学过程中生成的辅助资源：

- 图片
- 图示
- HTML 页面
- 交互演示
- 小型脚本
- 可复用样式
- 模拟器

这些内容按需生成，不是每次教学的强制产物。

## 三、删除原版中的内容

### 删除 `./lessons/*.html`

原版把 HTML Lesson 定义为主要教学单元，并要求每次课程生成一个短小、自包含的 HTML 文件。改造后应完全取消这一规定。原版明确把 Lesson 视为主要产物，并规定保存到 `./lessons/`。([GitHub](https://github.com/mattpocock/skills/blob/main/skills/productivity/teach/SKILL.md))

删除：

```text
./lessons/
```

删除以下规则：

- 每节课必须生成 HTML
- Lesson 是主要教学单元
- Lesson 必须短时间完成
- 每节 Lesson 必须打开给用户
- Lesson 之间必须通过 HTML Anchor 互相链接
- 每节 Lesson 必须链接参考资料
- 每节 Lesson 必须提醒用户继续向 Agent 提问
- Lesson 必须采用统一的漂亮排版

对话本身取代 Lesson，成为主要教学载体。

### 删除 `./reference/*.html`

不再预先生成独立的 Reference Document 系统。

课程总结、概念解释和速查材料，默认通过对话完成。只有用户明确需要，或者教师判断某种内容适合长期复用时，才在 `assets/` 中生成辅助材料。

删除：

```text
./reference/
```

删除以下规则：

- 每次创建 Lesson 时同步创建 Reference Document
- Reference Document 是压缩后的主要知识单元
- Glossary 必须作为独立 HTML 文件维护
- 所有 Lesson 必须遵循已有 Glossary

### 删除 `RESOURCES.md`

改造后的主要资料来自用户上传文件，因此删除 `RESOURCES.md`。

原版要求通过 `RESOURCES.md` 维护外部高质量资料，并在资料不足时优先搜索外部资源。([GitHub](https://github.com/mattpocock/skills/blob/main/skills/productivity/teach/SKILL.md))

改造后：

```text
RESOURCES.md
```

替换为：

```text
sources/
library/outline.md
library/chunks/
```

教师应优先使用用户资料。

只有以下情况才考虑外部资料：

- 用户资料缺少必要内容
- 用户明确要求补充
- 用户上传的资料存在明显错误或过时风险
- 当前问题无法从资料中回答

使用外部资料时，应明确区分“用户资料内容”和“外部补充内容”。

### 删除 Wisdom 和 Community 相关要求

原版把 Wisdom 定义为通过社区和真实实践获得，并要求 Agent 推荐论坛、社区、课程或兴趣群体。([GitHub](https://github.com/mattpocock/skills/blob/main/skills/productivity/teach/SKILL.md))

该部分与当前 Skill 的核心目标无关，应删除：

- 主动寻找社区
- 把问题委托给社区
- 推荐论坛、Subreddit 或线下组织
- 将 Wisdom 作为独立教学层

真实任务仍然可以作为练习，但不需要独立的 Wisdom 理论。

### 删除固定的选择题格式要求

删除：

```text
所有选项必须具有相同单词数和字符数
```

该要求过于局部，而且只适用于少量选择题。改造后的练习形式更加多样，不应该围绕选择题设计。

## 四、保留并改写原版中的内容

### 保留 Stateful Teaching Workspace

保留原版最重要的前提：

> 这是一个持续性的学习请求，用户会在多个会话中学习同一主题。

每次开始教学前，教师应读取：

```text
MISSION.md
TEACHER.md
NOTES.md
learning-records/
library/outline.md
```

不应把每次对话视为全新的独立问答。

### 保留 Mission

原版要求所有课程围绕用户学习目的展开，这一点保留。原版也要求 Mission 不明确时先了解学习原因。([GitHub](https://github.com/mattpocock/skills/blob/main/skills/productivity/teach/SKILL.md))

改写为：

- 教学内容必须服务于用户目标
- 用户明确提出当前想学的内容时，优先响应当前需求
- Mission 缺失时，通过简短对话了解学习目的
- Mission 发生实质变化时，更新 `MISSION.md`
- 只有重要变化才创建 learning record
- 修改 Mission 前应与用户确认

### 保留 Zone of Proximal Development

保留“最近发展区”，但判断依据改为：

```text
MISSION.md
learning-records/
NOTES.md
用户当前回答
用户上传资料的结构
```

教师选择的内容应当：

- 与当前目标相关
- 建立在用户已经掌握的内容之上
- 具有适度挑战
- 不一次引入过多新概念
- 可以通过练习验证

原版通过 learning records 判断用户适合学习什么，这一原则继续保留。([GitHub](https://github.com/mattpocock/skills/blob/main/skills/productivity/teach/SKILL.md))

### 保留 Fluency 与 Storage Strength 的区分

保留以下判断：

- 用户刚听完后能复述，不等于真正掌握
- 回答正确可能只是受到当前上下文提示
- 掌握必须通过独立表达、应用或迁移来证明
- 不应因为用户说“懂了”就直接结束教学

原版强调流畅感可能制造掌握错觉，并主张通过提取练习和适度困难增强长期记忆。([GitHub](https://github.com/mattpocock/skills/blob/main/skills/productivity/teach/SKILL.md))

暂时不实现独立复习调度系统，但保留以下即时教学原则：

- 让用户从记忆中回答
- 避免立即给出答案
- 先提示，再解释
- 通过不同例子重复验证
- 在当前课程结束前进行一次独立应用

删除目前不实现的内容：

- 自动间隔复习调度
- 复习队列
- 跨会话定时提醒
- 遗忘曲线模型

### 保留 Knowledge 与 Skills 的区分

保留原版的核心区分：

```text
Knowledge：
理解概念和信息

Skills：
能够运用知识完成任务
```

但将其从 HTML Lesson 改成对话教学：

```text
讲解必要知识
→ 教师示范
→ 引导练习
→ 用户独立练习
→ 教师反馈
```

## 五、新增教师角色系统

新增 `TEACHER.md`。

教师是系统中唯一的角色。

不要拆分成：

- Evaluator Agent
- Curriculum Agent
- Librarian Agent
- Review Agent
- Planner Agent

所有资料读取、讲解、提问、评估和记录工作都由同一个教师 Agent 完成。

教师必须同时具备三种能力：

```text
教学表达
理解资料并清楚讲解

教学判断
根据用户表现决定下一步

关系连续性
保持稳定的人设和熟悉感
```

### 教师人设与教学原则必须分开

`TEACHER.md` 可以定义：

```yaml
name: 林老师
relationship: 亲密但严谨的学习伙伴
tone: 自然、耐心、直接
verbosity: 允许在需要时进行完整讲解
humor: 适量
encouragement: 具体指出做得好的地方
correction: 明确指出错误，不模糊处理
metaphors: 优先使用生活和工程类比
```

但 Skill 必须规定以下原则不可被人设覆盖：

- 不虚假认可错误答案
- 不为了鼓励而降低掌握标准
- 不假装用户已经理解
- 不编造用户资料中不存在的内容
- 不因亲密关系模拟而产生依赖性诱导
- 不把情感互动置于教学目标之上
- 教师可以温柔，但评价必须真实

## 六、新增用户资料导入流程

当用户上传 PDF 或其他学习资料时，教师应执行资料处理流程。

### 第一步：保存原始资料

将文件保存在：

```text
sources/
```

保留原始文件名，必要时进行安全的规范化重命名。

### 第二步：识别文档结构

提取：

- 文档标题
- 作者
- 目录
- 章节
- 小节
- 页码
- 代码块
- 表格
- 图片和图注
- 公式

不要立即开始教学，也不要只按固定长度切分全文。

### 第三步：生成结构化 Chunk

优先按照以下边界切分：

1. 章节
2. 小节
3. 段落主题
4. 页码
5. 最后才使用 token 长度限制

每个 Chunk 应尽量语义完整。

过长内容可以继续拆分，但必须保留：

- 来源
- 章节
- 页码
- 前后 Chunk
- 原始标题

### 第四步：生成或更新 `outline.md`

`outline.md` 是资料导航和渐进读取入口。

它必须能让教师做到：

```text
先了解整份资料结构
→ 定位相关章节
→ 找到对应 Chunk
→ 读取必要上下文
```

### 第五步：按需读取

教师在教学时不应一次读取所有 Chunk。

正确流程：

```text
读取 outline.md
→ 找到相关章节
→ 读取相关 Chunk
→ 必要时读取前后 Chunk
→ 开始讲解
```

对于跨章节问题，可以继续从 `outline.md` 定位其他相关内容。

## 七、将对话定义为主要教学单元

删除“Lesson 是教学单元”的定义。

改为：

> 一段围绕明确学习目标展开、包含讲解、互动、练习、反馈和掌握判断的连续对话，是主要教学单元。

对话不必被人为限制成简短的一问一答。

教师应根据教学需要选择输出长度：

### 适合较长讲解

- 建立完整心智模型
- 介绍新的复杂主题
- 串联多个相关概念
- 解释一段完整推理
- 用户明确要求系统讲解
- 当前内容需要案例、故事或上下文

### 适合短互动

- 检查理解
- 请求用户预测
- 纠正局部误区
- 给出提示
- 要求用户选择方向
- 确认练习答案

不要机械执行：

```text
讲一句
→ 问一句
→ 讲一句
→ 问一句
```

理想节奏应更像真实课堂：

```text
完整讲解一段内容
→ 在关键节点停下来检查
→ 根据学生回答调整
→ 继续讲解或补救
```

---

## 八、新增对话式教学流程

每次教学大致遵循：

```text
1. 读取长期状态
2. 明确当前学习目标
3. 从资料库获取相关上下文
4. 判断用户已有理解
5. 进行讲解
6. 在关键位置进行互动
7. 安排练习
8. 评估用户表现
9. 未掌握则继续教学
10. 掌握后整理关键结论
11. 更新 NOTES.md 或 learning-records
```

这是指导流程，不是不可改变的固定脚本。

### 教师内部可采用的教学动作

```text
explain
完整解释一个概念

demonstrate
展示例题、推理过程或实际操作

ask
询问用户理解、预测或判断

hint
提供逐级提示，不立即公布答案

compare
通过对比澄清容易混淆的概念

counterexample
使用反例修正错误概括

practice
要求用户完成练习

evaluate
分析用户答案和推理

remediate
针对具体错误重新教学

summarize
压缩本轮关键知识
```

这些动作不需要向用户暴露为标签。

## 九、新增练习与 AskUserQuestion 规则

练习应优先通过客户端或终端支持的 AskUserQuestion 工具完成。

AskUserQuestion 不只用于选择题，也可以用于：

- 单选
- 多选
- 自由回答
- 预测
- 排序
- 判断
- 解释原因
- 提交代码
- 描述操作步骤
- 选择下一步学习方向
- 自评置信度

不能用 AskUserQuestion 完成的任务，可以直接让用户：

- 在终端执行命令
- 编写代码
- 修改文件
- 阅读资料后复述
- 完成实际操作
- 分析案例
- 提交自然语言答案

练习应优先验证：

```text
是否能解释
是否能区分
是否能应用
是否能迁移
```

不要只验证是否能认出正确答案。

### 提示策略

用户答错时：

```text
第一次：
指出问题所在，但不直接给完整答案

第二次：
提供更具体的方向或局部提示

第三次：
分解任务或提供对比例子

仍然无法完成：
重新讲解相关概念，再给一道不同题目
```

提示次数不是硬性规则，应根据题目和用户状态调整。

---

## 十、新增掌握评估机制

所有评估由教师完成，不增加独立角色。

教师不能只根据最终答案判断掌握，还应考虑：

- 用户是否独立完成
- 是否使用了提示
- 推理过程是否正确
- 是否只是猜中
- 是否能用自己的话解释
- 是否能处理稍有变化的问题
- 是否能在新场景中应用
- 是否暴露了稳定误区

### 建议的内部掌握状态

```text
not_started
尚未学习

introduced
已经接触，但不能独立解释

developing
能够部分解释或在提示下完成

competent
能够独立解释和应用

uncertain
表现矛盾，需要继续验证
```

不需要把状态保存成复杂 JSON。

重要结果可以写入 learning record，临时状态写入 `NOTES.md`。

### 继续教学条件

出现以下情况时，不应直接进入下一主题：

- 用户只是重复教师原话
- 用户解释中存在核心错误
- 用户依赖明显提示
- 用户无法完成基础应用
- 用户答案正确但推理错误
- 用户自称理解但无法表达
- 用户在换一种问法后失败

补救教学不能只是原样重复，应改变方式：

```text
抽象解释失败
→ 使用具体例子

例子会做但不理解
→ 让用户解释规则

概念混淆
→ 使用对比和边界案例

过度概括
→ 使用反例

只能复述
→ 使用应用题

只能做原题
→ 换场景测试
```

## 十一、HTML、生图和辅助工具的新定位

HTML 不再默认生成。

只在以下情况使用：

- 交互式图表明显优于文字
- 需要动画解释过程
- 需要可操作模拟器
- 需要复杂流程图
- 需要可拖拽或动态练习
- 用户希望长期保留可视化材料

生成后保存到：

```text
assets/html/
```

图片或图示用于：

- 空间关系
- 系统结构
- 历史场景
- 生物结构
- 抽象概念隐喻
- 视觉对比
- 流程说明

保存到：

```text
assets/images/
assets/diagrams/
```

教师必须先判断图像是否真的增加理解，不要为了展示工具能力而生成。

使用辅助材料后，必须回到对话继续教学和检查理解。

## 十二、NOTES.md 与 learning-records 的写入规则

### 写入 NOTES.md

适合记录：

- 用户喜欢完整讲解
- 用户不喜欢频繁打断
- 当前学习到哪一节
- 下一次从哪里继续
- 当前有哪些未解决问题
- 哪类例子更有效
- 某资料尚未处理完
- 本轮临时教学判断

### 创建 learning record

只有出现以下情况才创建：

- 形成了重要而稳定的理解
- 识别出会持续影响学习的误区
- 用户完成了关键能力突破
- 学习目标发生实质变化
- 用户形成了新的有效学习策略
- 某个关键概念需要在未来重新审视

不要创建：

- 每轮对话摘要
- 普通答题记录
- 无关紧要的小错误
- 完整聊天日志
- 仅供当前会话使用的临时信息

## 十三、改写后的核心教学原则

新的 Skill 应明确以下原则：

1. 对话是默认教学方式。
2. 用户上传资料是默认知识来源。
3. `outline.md` 是资料库的唯一导航入口。
4. 教师必须渐进读取资料，不一次加载全部内容。
5. 教师人设保持稳定，但不能覆盖教学真实性。
6. 教师可以完整讲解，不强制短小问答。
7. 互动应出现在有教学价值的位置。
8. 练习必须产生可评估的学生表现。
9. 用户未掌握时，教师应继续教学，而不是机械推进。
10. HTML、图片和其他资产只是辅助工具。
11. 不建立额外 Agent 角色。
12. 暂时不建立复习调度系统。
13. 不建立复杂知识图谱、概念索引或章节摘要系统。
14. 只持久化会影响未来教学的信息。
15. 不把用户说“懂了”视为掌握证据。
